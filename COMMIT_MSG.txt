feat(track-c): headway control + bunching alerts

Adds the traffic-jam mitigation layer recommended by TransitCenter /
Metro Magazine / APTA BRT design guidance — purely software, no signal
or lane infrastructure required.

Schema (migration 014_headway_control.sql):
- routes.target_headway_min  — NULL disables headway control.
- alert_type 'bus_bunching'  — new enum value.
- headway_observations       — append-only log, tenant-RLS'd.
- RPC route_headway_status(p_operator) → per-route gauge data.
- RPC detect_bunching(vehicle_id, lat, lon, threshold_m) → closest
  same-route vehicle and recommended hold_seconds (uses an 18 km/h
  Damascus dense-traffic median speed).
- Seed: R101/R102 target headway 8min, R201/R202 target 5min.

API:
- GET /api/admin/headway (dispatcher+) — wraps route_headway_status.
- _bunching_check() runs after every /api/driver/position write,
  dedupes alerts (no repeat within 5 min for the same pair), records
  every observation, and surfaces hold_seconds + gap_m +
  other_vehicle_id on the position response.

UI:
- /driver/ amber hold banner with live countdown when hold_seconds > 0
  (cooperative — no forced behavior, just a clear suggestion).
- /admin/dispatch.html new headway strip above the trips table; one
  card per active route showing target vs actual headway with green /
  red / amber / grey status; refreshes on the existing 20s timer.

48 endpoints total. AST parse OK. See FIXES_APPLIED.md "Track C" for
the full per-file changelog.
